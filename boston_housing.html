Implementation: Define a Performance Metric¶

It is difficult to measure the quality of a given model without quantifying its performance over training and testing. This is typically done using some type of performance metric, whether it is through calculating some type of error, the goodness of fit, or some other useful measurement. For this project, you will be calculating the coefficient of determination, R2, to quantify your model's performance. The coefficient of determination for a model is a useful statistic in regression analysis, as it often describes how "good" that model is at making predictions.

The values for R2 range from 0 to 1, which captures the percentage of squared correlation between the predicted and actual values of the target variable. A model with an R2 of 0 is no better than a model that always predicts the mean of the target variable, whereas a model with an R2 of 1 perfectly predicts the target variable. Any value between 0 and 1 indicates what percentage of the target variable, using this model, can be explained by the features. A model can be given a negative R2 as well, which indicates that the model is arbitrarily worse than one that always predicts the mean of the target variable.

Question 3 - Training and Testing¶

  • What is the benefit to splitting a dataset into some ratio of training and testing subsets for a learning algorithm?

Hint: Think about how overfitting or underfitting is contingent upon how splits on data is done.

Answer: Having fenced off testing data ensures that there is a methodology available for checking fairly for fit, and in particular, overfitting. A model may look good, in that it fits the training data well (eg high R2), but actually be seriously overfitted, and therefore not generalise well. Overfitting can be diagnosed by a poor ability to fit the testing data versus a strong training data performance (perhaps suspiciously strong). Overfitting can be difficult to spot without this test.

If too little data is put aside for testing, the evaluation metrics may not be particularly reliable. On the other hand, if too much data is put aside for testing, the model may not be as accurate as it could be were it given a bigger dataset on which to train, and in particular may be underfitted as it may not have seen examples of the full range of the data. Around 20% is generally considered a good trade-off.

Question 7 - Grid Search¶

  • What is the grid search technique?
  • How it can be applied to optimize a learning algorithm?

Hint: When explaining the Grid Search technique, be sure to touch upon why it is used, what the 'grid' entails and what the end goal of this method is. To solidify your answer, you can also give an example of a parameter in a model that can be optimized using this approach.

Answer: The Grid Search technique is an automated and efficient way of searching across combinations of values of model hyper-parameters (such as maximum depth of a decision tree, or gamma and C in a support vector machine model) in a much more efficient and thorough fashion than could be achieved manually, to achieve optimisation of the model. The end goal is to find the "best" model, ie the model that best fits the test data, avoiding the traps of both underfitting and overfitting.

The technique can be applied by putting aside a chunk of the training data set for cross-validation purposes (not touching the test set), training a model on that data under each of the possible combinations of parameters across plausible ranges, and selecting the model that performs best on the cross-validation sample.

It is important to note that the "test" sample needs to be kept in reserve until after the final model has been chosen.

Question 8 - Cross-Validation¶

  • What is the k-fold cross-validation training technique?

  • What benefit does this technique provide for grid search when optimizing a model?

Hint: When explaining the k-fold cross validation technique, be sure to touch upon what 'k' is, how the dataset is split into different parts for training and testing and the number of times it is run based on the 'k' value.

When thinking about how k-fold cross validation helps grid search, think about the main drawbacks of grid search which are hinged upon using a particular subset of data for training or testing and how k-fold cv could help alleviate that. You can refer to the docs for your answer.

Answer: The k-fold cross-validation technique involves dividing the training data into k random subsamples of equal sizes (or thereabouts). 10 is a commonly used setting, but the variable k can generally be higher when there is more data. Some data has already been put completely aside for the final testing of the selected model.

In each round, one of the k subsets is temporarily put aside for validation (can be thought of as a temporary test set) and the remaining (k-1) subsamples are combined as the training data set over which a hyper-parameter grid search can be performed as above. It is important that the grid search process is repeated for each round, as re-using the best model from say the first round to save time would introduce an optimistic bias to the error measurement, as some of the training data will inevitably be reused as the cross-validation set for the remaining rounds. Even worse would be using cross-validation techniques after using the full training data set to identify key features, such as fixing one or more hyper-parameters.

After k-1 rounds, the error results are averaged and the model with the lowest average error is selected and tested on the true test set. If the fit on the test data set is significantly different from the inner cross validation then this is an indication that there may be overfitting. This can easily occur with hyper-parameter optimisation. Instability in the different estimates of the optimised parameters is also a bad sign that can probably only really be fixed with more data.

The benefit the k-fold cross validation technique provides for grid search when optimising a model is that the average estimation across a bunch of randomly selected subsamples will be more robust to data quirks that may be present in any one sub-sample used for either training or cross-validation. Having cross-over in the data samples ensures there is sufficient data for each estimation even in the case of relative small samples, and does not bias the results as long as the grid search process is repeated each time.

# TODO: Import 'make_scorer', 'DecisionTreeRegressor', and 'GridSearchCV'
from sklearn.metrics import make_scorer
from sklearn.tree import DecisionTreeRegressor
from sklearn.grid_search import GridSearchCV

def fit_model(X, y):
    """ Performs grid search over the 'max_depth' parameter for a 
        decision tree regressor trained on the input data [X, y]. """
    
    # Create cross-validation sets from the training data
    # sklearn version 0.18: ShuffleSplit(n_splits=10, test_size=0.1, train_size=None, random_state=None)
    # sklearn versiin 0.17: ShuffleSplit(n, n_iter=10, test_size=0.1, train_size=None, random_state=None)
    cv_sets = ShuffleSplit(X.shape[0], n_iter = 10, test_size = 0.20, random_state = 0)

    # TODO: Create a decision tree regressor object
    regressor = DecisionTreeRegressor(max_depth=None)

    # TODO: Create a dictionary for the parameter 'max_depth' with a range from 1 to 10
    params = {"max_depth":range(1,11)}

    # TODO: Transform 'performance_metric' into a scoring function using 'make_scorer' 
    scoring_fnc = make_scorer(performance_metric)

    # TODO: Create the grid search cv object --> GridSearchCV()
    # Make sure to include the right parameters in the object:
    # (estimator, param_grid, scoring, cv) which have values 'regressor', 'params', 'scoring_fnc', and 'cv_sets' respectively.
    grid = GridSearchCV(estimator = regressor, param_grid = params, scoring = scoring_fnc, cv = cv_sets)

    # Fit the grid search object to the data to compute the optimal model
    grid = grid.fit(X, y)

    # Return the optimal model after fitting the data
    return grid.best_estimator_

Sensitivity¶

Answer: No, this model should not be used in a real world setting. Some problems with it in this context:

  • Scaling all the prices by a simple constant to account for inflation will not take into account that suburbs will have become more or less fashionable over that time. And the inflation measure may not be the right one.
  • There are many other features of a home that are missing from the model relevant, eg general condition, parking/garaging, swimming pool, size of section, year built, amount of sunshine... This will lead to omitted variable bias.
  • The model does not appear to be particularly robust, in that the estimates of the value of a client's home varied by almost 70k with different samples.
  • At best, the results are applicable only to Boston and cannot be applied to anywhere else.
  • No, it is not reasonable to judge the price of an individual home based on the characteristics of the entire neighbourhood. Prices are certainly correlated over a neighbourhood but the features of the individual house matter hugely too.
